# AlembicDesign 运行规则

更新日期：2026-05-25

## 目的

`AlembicDesign` 将需求讨论与实现派发拆开。它给用户一个专门讨论想法、打磨需求和比较方案的窗口；`AlembicWorkspace` 仍然是执行、验收、TODO 归口和归档的总控。

## 控制关系

- `AlembicWorkspace` 负责最终状态、当前主线、TODO 优先级、窗口派发、测试交接、验收和归档。
- `AlembicDesign` 负责探索性需求讨论和设计草案准备。
- 设计草案只有在 `AlembicWorkspace` 接收并创建对应 workspace 确认 / wave 文档后，才会变成可执行工作。
- 如果 `AlembicDesign` 被单独打开且读不到父级 workspace 文档，只能进入 `detached-design-mode`：草案可以继续写，但必须标注“未接入总控当前状态，等待 AlembicWorkspace 导入复核”。

## 允许事项

- 澄清产品目标和用户场景。
- 起草原始计划书和需求设计。
- 比较架构方案和取舍。
- 识别需要 `AlembicWorkspace` 或源码仓库窗口补充的代码调研请求。
- 识别讨论内容是否应作为 bug 线索、TODO 候选、调研请求、用户决策或当前主线阻塞交回总控。
- 产出 workspace 交接草案。
- 维护设计窗口模板和本地设计文档。
- 当工程设计出现复杂分叉、用户经验可能直接改变方案时，直接向用户提出简短确认问题，不绕远路。

## 与总控需求设计能力对齐

`AlembicDesign` 的设计产物必须能被 `AlembicWorkspace` 直接接入成熟路线：原始计划书 → 用户确认 → 需求设计 → 代码实现依赖调研 → 目标阶段确认 → 用户确认 → wave。它不需要复制总控的派发和验收能力，但必须把前置设计信息准备到足够完整。

对齐规则：

- 原始计划书只记录用户目标、背景、范围、约束和确认问题；用户确认前，不写执行阶段、不推荐发送窗口。
- 需求设计必须包含用户场景、完整功能闭环、输入输出、状态变化、生产方、消费方、失败路径、仓库边界、验证策略和完成定义。
- 复杂需求必须显式判断是否需要代码实现依赖调研；若设计窗口没有足够代码证据，必须把调研范围、入口、调用链和待验证问题写成交接给总控的请求。
- 阶段只能作为候选方向，不能作为执行派发依据；最终阶段顺序由总控在目标阶段确认里确定。
- 讨论中产生的 TODO、风险、删除候选、兼容保留、验证缺口和用户偏好，必须落到设计文档或交接草案，不能只留在聊天里。
- 任何会降低目标能力、只做框架、延期关键能力、保留兼容层或改变仓库职责边界的设计，必须标为待确认。
- 每个需求都必须写清与总控当前主线的关系：不影响当前主线、TODO 候选、下一主线候选、当前主线阻塞，或需要总控确认。默认不得因为设计讨论打断当前实现主线。
- 对不需要完整需求设计的内容，使用轻量 signal 回传总控；signal 可以随时产生，不受当前需求文档进度影响。

## 禁止事项

- 修改产品源码。
- 直接执行真实项目测试。
- 直接跨仓库派发任务。
- 作为权威来源修改 `AlembicWorkspace` 当前状态。
- 将密钥、私有 token 或用户机器特定信息写入长期文档。

## 交接契约

`AlembicDesign` 与 `AlembicWorkspace` 的交流分两类：

- **Signal**：用于 bug 线索、TODO 候选、调研请求、用户决策、当前主线风险或轻量建议。使用 `templates/workspace-signal-template.md`，总控接收后决定是否进入全局 TODO、当前 wave、测试单或新主线。
- **Handoff**：用于较完整的需求设计或方案交接。使用 `templates/workspace-handoff-template.md`，总控接收后决定是否继续原始计划确认、需求设计、代码调研、目标阶段确认或 wave。
- **Handoff board**：正式需求设计完成后的清单入口，固定为 `docs/current/workspace-handoff-board.md`。Design 把准备交给总控的条目标为 `ready-for-workspace`；总控脚本自动读取该清单并生成 workspace 收件箱。清单登记不等于 workspace TODO 入账，也不等于派发。

每次交回 `AlembicWorkspace` 的 handoff 草案应包含：

1. 需求标题和用户目标。
2. 设计状态：草案、用户已确认、阻塞、准备交给 workspace 评审。
3. 最终完成定义。
4. 已知事实和证据。
5. 开放问题和已确认决策。
6. 建议覆盖的仓库 / 窗口。
7. 建议阶段顺序。
8. 验证需求。
9. 非目标和禁止捷径。

`AlembicWorkspace` 接收 handoff 前仍需独立复核；handoff 不是目标阶段确认，也不是 wave 分派表。

Signal / handoff 可以随时交回总控；清晰文档路线优先于聊天口头记忆。总控主线是否因此改变，只由 `AlembicWorkspace` 判断。

完整需求的默认路线是 handoff board，而不是反复发送轻量 signal。轻量 signal 只处理 bug、当前主线风险、用户决策或临时调研发现。

## Signal 判断规则

讨论内容进入 signal 时，先判断一类主类型：

- `bug`：用户描述异常、失败、回归、体验问题或“看起来不对”。
- `todo`：用户提出可后续处理、可并行或暂不打断主线的事项。
- `research`：需要补代码事实、外部调研或方案比较才能判断。
- `decision`：用户明确偏好、边界、非目标或长期路线。
- `current-mainline-risk`：可能改变当前主线完成定义、阻塞点或验证方式。
- `requirement-candidate`：可能成为下一条完整需求主线，但还未进入原始计划。

Signal 必须写清：触发原话 / 摘要、判断类型、影响范围、证据状态、推荐归口、是否建议打断当前主线、总控下一步建议。

## 总控接收门禁

交给 `AlembicWorkspace` 前，设计窗口必须确认：

- 原始计划是否已经用户确认；未确认时只能建议“进入原始计划确认”，不能建议执行。
- 需求设计是否包含完整功能闭环、仓库边界、验证策略和完成定义。
- 代码事实是否足够；不足时明确列出调研入口和待验证调用链。
- 阶段是否仍为候选，没有写成“发送给某窗口执行”。
- TODO / Backlog 是否保留了讨论中的风险、验证缺口、用户偏好和后续拆分输入。
- 是否存在需要用户确认的删减、降级、延期、兼容保留或边界变更。
- signal 是否足够轻量；如果已经需要完整功能闭环和阶段候选，应升级为 handoff 或需求设计。

交接草案必须清楚到 `AlembicWorkspace` 不需要重放整段讨论，也能判断下一步。

## 设计质量检查卡

交付设计前，逐项检查：

1. 用户目标和最终完成定义是否清楚。
2. 需求是否形成完整功能闭环，而不是抽象连接或空壳接口。
3. 生产方、消费方、输入、输出、状态变化和失败路径是否明确。
4. 每个 Alembic 相关仓库是参与、观察、无任务，还是待总控调研。
5. 已知代码事实是否和证据来源分开记录；缺证据时是否写为调研缺口。
6. 是否判断外部调研需求，并说明理由。
7. 阶段是否仍是候选，没有伪装成可执行 wave。
8. TODO / Backlog 是否记录了讨论中发现的风险、验证点和用户偏好。
9. 是否存在需要用户或总控确认的删减、降级、兼容、删除或边界变化。
10. handoff 是否明确写清当前主线关系和总控接收建议。

## 当前主线意识

默认情况下，`AlembicDesign` 不打断 `AlembicWorkspace` 当前主线。若实现主线进行中出现新想法，`AlembicDesign` 可以先起草，并标记它属于：

- 背景讨论；
- bug 线索；
- TODO 候选；
- 调研请求；
- 用户决策；
- 下一主线候选；
- 当前主线紧急阻塞。

只有 `AlembicWorkspace` 能把它提升为当前执行主线。
